SynthDefs 


Utiliza la clase Synthdef para construir sintetizadores que van a ser ejecutados en el servidor. Los sintetizadores, que pueden ser guardados en el disco y reusados, son parecidos a los 'presets' en los sintetizadores de hardware y software comerciales .


Al ser escritos como código en los programas del cliente, los sintetizadores tienen dos partes clave: un nombre y una función. En el lenguaje técnico de SuperCollider, la función tiene el nombre de ugenGraphFunc.


El término ugenGraphFunc se deriva de la noción de gráfica, que es una estructura de datos que SuperCollider utiliza para organizar los generadores de unidades. SuperCollider construye la gráfica para ti a partir del código que encuentra en tu función, lo que implica que no tienes que saber cómo es que una gráfica funciona, ni que existe siquiera.


Si quieres saber más de gráficas, visita la Wikipedia en:a at 


http://en.wikipedia.org/wiki/Graph_(data_structure). 


O, ve a


http://www.nist.gov/dads/HTML/graph.html

Template (plantilla)


Aquí hay un 'template' (plantilla) para un synthdef mostrando que consiste de un nombre y de una función ugenGraphFunc


SynthDef(

"aSynthDef", // el 1er argumento es el nombre

{ .... i am a ugenGraphFunc ... } // el 2o argumento es la función ugenGraphFunc

)


Para hacer que la plantilla sea funcional


1. escribir código dentro de la función ugenGraphFunc

2. mandar un mensaje '.load' al synthdef


(

SynthDef(

"aSynthDef", // el nombre del synthdef

{ // la función ugenGraphFunc con el código

arg freq;

Out.ar(

0,

SinOsc.ar(freq, 0, 0.1)

)

}

).load(s);

)


////////////////////////////////////////////////////////////////////////////////////////////////////


El mensaje '.load' y la variable 's'


El mensaje '.load' escribe los synthdefs en el disco duro y además los manda al servidor default. El servidor default está definido por SuperCollider al inicio (como el servidor localhost). En el  momento de ser iniciado también le es asignada la variable 's'.


////////////////////////////////////////////////////////////////////////////////////////////////////


El mensaje '.send' y el servidor remoto


El mensaje '.send', por otro lado,


SynthDef( .... ).send(s);


en vez del mensaje '.load'


SynthDef( .... ).load(s);


es otra manera de mandar un synthdef al servidor. El mensaje '.send', a diferencia del mensaje '.load', no escribe primero el synthdef al disco; sino que sólo lo transmite directamente al servidor. Este es, por lo tanto, el mensaje a utilizar para definir un synthdef en una computadora y mandarlo a otra.

(

var aServer;

aServer = 

Server(

"aRemoteServerOnAnotherMachine", 

NetAddr("... an ip # ...", 57110) // un servidor en otra máquina

);

SynthDef( .... ).send(aServer);

)

Navegador de SynthDef


Utiliza el navegador de synthdefs para examinar los synthdefs que han sido escritos en el disco-


(

// un navegador de synthdefs:

SynthDescLib.global.read;

SynthDescLib.global.browse;

)


La caja del centro (en la fila superior) muestra los nombres de los synthdefs. Cada nombre, al ser seleccionado, causa que las otras cajas muestren los 'Ugens', controles y entradas y salidas para un synthdef en particular.


La caja titulada 'SynthDef controls' es útil ya que muestra los argumentos que pueden ser enviados a un synthdef en particular.


El navegador muestra que el synthdef definido antes - "aSynthDef" - está compuesto por cuatro 'Ugens', un control, sin entradas, y una salida. Los cuatro 'Ugens' incluyen instancias de las clases 'Control', 'SinOsc', 'BinaryOpUgen' y 'Out'.


El único control es "freq". Un control es un argumento que un sintetizador puede utilizar al ser creado o en cualquier momento mientras existe en el servidor. El navegador también muestra que "aSynth" no tiene entradas (lo que significa que no utiliza datos de audio o buses de control) y que manda un canal de audio a la salida a un Bus de audio.
