Tipos de evento


Una pregunta frecuente es ¿Qué parámetros tienen un significado especial en Pbind? La respuesta sorprendente es que ¡ninguno de ellos lo tiene! Esto es debido a que Pbind simplemente puebla de datos el evento resultante; no le importa cuáles son esos datos.


El prototipo de evento usado cuando se reproduce un patrón define las acciones que tomará y es aquí donde se definen los parámetros. La mayoría de los patrones tocarán usando el prototipo de eventos predefinido (Event.default), así que esta es la fuente de los parámetros que se usarán en los casos más típicos. 


El prototipo predeterminado de eventos define un número de "tipos de evento", cada uno de los cuales ejecuta una tarea diferente. La llave \type ('tipo') determina la acción a tomar y los parámetros significativos dependerán del tipo de evento.


Hay muchos tipos de eventos. Sin embargo, solo unos cuántos son de uso común. El evento tipo \note es por mucho el más típico. Los otros son auxiliares, útiles cuando se escriben patrones para generar partituras (Score) adecuados para síntesis fuera de tiempo-real.


Antes de ver los tipos de evento, repasemos los parámetros estándar usados a través de muchos de ellos. (No todos los parámetros comunes son usados por cada tipo de evento, pero aparecen en muchos lugares). 


Parámetros comunes


Control de temporalización


\delta: Número de pulsos hasta el siguiente evento. Calculado a partir de  ~dur * ~stretch, si \delta no es dado explícitamente. 

\lag: Número de segundos para retrasar los mensajes que el evento manda al servidor.

\timingOffset: Número de pulsos para retrasar los mensajes que el evento manda al servidor. En conjunción con Quant, esto permite control sobre el orden de preparación de eventos entre distintos patrones en el cliente, sin que se desincronicen los eventos sónicos que deban sonar al mismo tiempo. En GP_06g_Compartiendo_Datos  hay un ejemplo de su uso para pasar información de un patrón de bajo a uno de acordes. 


\sustain: Número de pulso para esperar antes de liberar un nodo de sinte en el servidor.  El SynthDef debe tener un argumento de compuerta (llamado 'gate') para que la la liberación sea enviada explícitamente; de otra manera, el patrón asume que las notas se liberarán a sí mismas usando un envolvente temporizado. \sustain es calculado a partir de  ~dur * ~legato * ~stretch si no es dado directamente.


Control de Nodos


\addAction: Cómo añadir un nodo de sinte o de grupo relativo al \group dado en el evento. Ver Synth.

\amp: Aunque no está definido formalmente como un parámetro especial, se usa típicamente para controlar la amplitud de los sintes. El SynthDef debe tener un argumento 'amp' y usarlo para controlar su volumen. \amp es opcionalmente calculado a partir de \db.

\id: Los identificadores (id) para la creación de nuevos Nodos en el evento. Normalmente esto en nulo, en cuyo caso los IDs serán obtenidos automáticamente de server.nextNodeID.

\instrument: El nombre del SynthDef para el cual los nodos serán creados. Solo un nombre debe ser dado a la vez (a diferencia de otros argumentos que se expanden en multicanales para crear nodos múltiples).

\group: El nodo que es el objetivo relativo al cual nuevos nodos serán creados. Similar a 'target' (objetivo) en Synth(nombreDeDefinición, argumentos, objetivo, acciónDeAñadir).

\out: Generalmente usado para la salida de un Sinte. Cuando se usa Pbus o Pfxb un canal de audio es asignado para aislar la señal del patrón. Todos los eventos del patrón reciben el nuevo número de canal en la llave \out y los SynthDefs que son tocados deben tener un argumento 'out' como objetivo de los UGens de salida, por ejemplo Out.ar(out, ... ).



Tipos de Evento


Control de Nodos


rest: (silencio, "descanso"). Como uno esperaría, un \rest no hace nada excepto esperar el tiempo requerido hasta el siguiente evento.


note: Este es el tipo predeterminado de evento, usado cuando \type no se especifica. Toca uno o más nodos de síntesis en el servidor, con liberación automática después de \sustain número de pulsos si el SynthDef tiene un argumento 'gate' (compuerta).

  • Argumentos estándares para el control de Temporalización y de Nodos
  • sendGate ("mandarCompuerta"): Sobrepasar el comportamiento del SynthDef para el argumento para el argumento 'gate'. Si el SynthDef tiene 'gate', ajustando sendGate = false previene que el mensaje de liberación sea enviado. Raramente usado.
  • strum ("rasgueo"): Si se producen múltiples notas (usualmente acordes, producidos al proveer un arreglo como uno de los parámetros de alturas), \strum es el número de pulsos a retrasar cada comienzo sucesivo de las notas. Cuando se usa \strum, otra llave se vuelve activa, \strumEndsTogether ("terminan juntos el rasgueo"). Si es falso (predeterminado), cada nodo con rasgueo (arpegiado) tocará durante su duración completa y las liberaciones estarán apiladas. Si es cierto, las liberaciones ocurrirán al mismo tiempo.


p = Pbind(

// el arreglo se expande en multicanales hacia un sinte para cada elemento

\degree, #[2, 5, 8, 11, 13, 16],

\octave, 3,

\dur, 2,

\sustain, 3,

// strum distinto a cero apila las entradas

\strum, 0.12

).play(quant: 2);


p.stop;


on (encender):  un nodo (o nodos) de Sinte sin liberación. Los ID(s) de los nodos están la variable ~id del evento. Esos ID's pueden ser usados con eventos tipo off (apagado), set (ajustar) o kill (matar).

  • Argumentos estándares para el control de Temporalización y de Nodos
  • callback ('llamada de regreso'): Una función que será ejecutada después de crear los nodos. Usarla para contabilizar, para mantener la pista de los IDs de nodo para uso posterior. La función recibe el evento finalizado como su argumento.
  • (los parámetros sendGate y strum no son usados)


off (apagar): Liberar los nodos en el servidor de buena manera, si es posible. Si el SynthDef tiene un argumento de compuerta 'gate', la compuerta se ajustará a 0 o un valor especificado por el usuario. Do otro modo, los nodos serán brutalmente asesinados con n_free.

  • Argumentos estándares para el control de Temporalización y de Nodos
  • hasGate (tieneCompuerta): Cierto o falso, diciéndole al evento si el SynthDef tiene o no argumento de compuerta 'gate'. El predeterminado se asume cierto. 
  • id: Los IDs de nodo deberán ser dados explícitamente.
  • gate (compuerta): Predeterminado en 0. Valores negativos disparan una "liberación forzada" en EnvGen. Ver página de ayuda de EnvGen para detalles.


kill (matar): Remover nodos inmediatamente usando n_free.

  • Argumentos estándares para el control de Temporalización y de Nodos
  • id: Los IDs de nodo deberán ser dados explícitamente.


set: Mandar nuevos valores a las entradas de control de nodos existentes.

  • Argumentos estándares para el control de Temporalización y de Nodos
  • id: Los IDs de nodo deberán ser dados explícitamente.


Hay dos maneras de especificar los nombres de argumentos: por instrumento y por arreglos de argumentos.  


Por instrumento:


  • instrument: El nombre del SynthDef debe ser dado otra vez, de manera que el evento sepa qué llaves-valores del evento son relevantes para los nodos.
  • args: Predeterminado, la llave \args contiene los nombres de control para el synthdef predeterminado(\default). Para tomar nombres de argumentos a partir del nombre de instrumento, uno debe reemplazar este predeterminado con un arreglo vacío (o cualquier objeto que no sea una colección).


(

SynthDef(\event_set, { |freq = 440, gate = 1, amp = 0.1, lagTime = 0.1,

ffreq = 2000, detune = 1.005, out = 0|

var sig = SyncSaw.ar(Lag.kr(freq, lagTime) * [1, detune], ffreq).sum * amp

* EnvGen.kr(Env.adsr, gate, doneAction: 2);

Out.ar(out, sig ! 2);

}).memStore;

)


a = Synth(\event_set);


(

p = Pbind(

\type, \set,

\instrument, \event_set,

// Si no se reemplaza \args, el parámetro de \ffreq del sinte no sería ajustado.

// Probar la diferencia con y sin la siguiente línea:

\args, #[],

\freq, Pexprand(200, 600, inf),

\ffreq, Pkey(\freq) * Prand([2,3,5,7], inf),

\dur, Pwhite(1, 5, inf) * 0.125

).play;

)



p.stop;

a.free;


Por nombres de argumentos: 


  • args: Proveer una lista de nombres de los argumentos de un Sinte, por ejemplo [\freq, \amp, \pan]. De esta manera no es necesario proveer el nombre del instrumento. 


a = Synth(\event_set);


(

p = Pbind(

\type, \set,

\args, #[\lagTime, \freq],

\lagTime, Pwhite(0.01, 1.5),

\freq, Pexprand(200, 600, inf),

\dur, Pwhite(1, 5, inf) * 0.125

).play;

)


p.stop;

a.free;


monoNote

monoOff

monoSet: Estos tipos de evento son usados internamente por Pmono y PmonoArtic. No deben ser usados directamente. 



Control del servidor


group: Crear un nuevo grupo (o grupos).

  • Argumentos estándares para el control de Temporalización y de Nodos
  • id: (opcional) IDs para los nuevos grupos. Si no se especifican, el nuevo ID (para un sólo grupo) puede ser encontrado en el evento después de .play. Para crear grupos múltiples se debe proveer un arreglo de IDs.


bus: Ajustar el valor de un canal de control o de canales contiguos. Se asume que uno ya tiene los índices de los canales. 

  • Argumentos estándares para el control de Temporalización y de Nodos
  • array: El/los valor(es) para mandar al/los canal(es). No tiene que ser un arreglo si es nada más un valor.
  • out: El primer índice de canal a ajustar. Se puede usar un objeto Bus.



Control de Buffers


Todos estos tipo de evento esperan que se provea un número de buffer. No tomarán automáticamente los números de buffer del asignador de buffers del servidor. Se permite un objeto Buffer -- uno podría crear el Buffer con Buffer.alloc o Buffer.new y luego usar este objeto en los eventos de control. Ver también la sección "Tipos de evento con limpieza" más abajo para otras maneras más amigables para el usuario con las cuales controlar opciones de los Buffers.


alloc(asignar): Asignar memoria para un buffer en el servidor. Solamente un buffer puede ser asignado por evento. 

  • Argumentos estándares para el control de Temporalización y de Nodos
  • bufnum, numchannels, numframes: Ver la página de ayuda de Buffer.


free: cancelar la memoria del buffer en el servidor. 

  • Argumentos estándares para el control de Temporalización y de Nodos
  • bufnum: El número de buffer a liberar (solamente uno).


gen: Generar datos para tablas de onda en un buffer, usando los plug-ins b_gen del servidor. La página de ayuda de Buffer tiene más detalles sobre los plug-ins. 

  • Argumentos estándares para el control de Temporalización y de Nodos
  • bufnum
  • gencmd: El nombre del plug-in generador: \sine1, \sine2, \sine3, \cheby.
  • genflags: Tres  señales asociadas a números: normalizar = 1, comoTablaDeOnda = 2, limpiarPrimero = 4. Añadir los números para las señales deseadas. Normalmente todas son ciertas, sumando hasta 7.
  • genarray: Parámetros de datos para el plug-in. Ver la página de ayuda Server-Command-Reference para detalles en los formatos de cada plugin. 


load: Asignar memoria de buffer en el servidor y cargarle un archivo de sonido usando b_allocRead. 

  • Argumentos estándares para el control de Temporalización y de Nodos
  • bufnum
  • filename: Ruta al archivo en el disco. 
  • frame: Primer cuadro a leer (predeterminado 0).
  • numframes: Número de cuadros a leer (predeterminado 0, que carga el archivo completo).


read: Leer una archivo de sonido en un buffer que ya ha sido asignado en el servidor. Este tipo de evento es buen para preparar un archivo de sonido para su uso con DiskIn.

  • Argumentos estándares para el control de Temporalización y de Nodos
  • bufnum
  • filename: Ruta al archivo en el disco. 
  • frame: Primer cuadro a leer del archivo (predeterminado 0).
  • numframes: Número de cuadros a leer (predeterminado 0, que carga el archivo completo).
  • bufpos: Cuadro de comienzo en el buffer (predeterminado 0).
  • leaveOpen(dejar abierto): 1 = dejar el archivo abierto (para uso con DiskIn). 0 = cerrar el archivo después de leerse. Predeterminador = 0.



Tipos de evento con limpieza


Estos tipos de eventos son únicos en el sentido de que tienen funciones de limpieza automática asociados a ellos. Reproducir uno de estos tipos de evento asigna recursos en el servidor. Después, los recursos pueden ser liberados al cambiar el tipo de evento a su correspondiente tipo de limpieza y reproduciéndolo otra vez. Mientras los recursos están activos, el evento puede ser utilizado como referencia a los recursos en otros evento o para mandar mensajes a Synth. 


// crear un buffer

b = (type: \allocRead, path: "sounds/a11wlk01.wav").play;


a = { PlayBuf.ar(1, b, doneAction: 2) }.play;


// remover buffer

EventTypesWithCleanup.cleanup(b);


El ejemplo de Pproto en GP_06f_Control_del_servidor, muestra como éstos se pueden usar para limpiar los objetos en el servidor al final de un patrón. 


audioBus: Asignar un índice de canal de audio en el servidor. 

  • channels: Número de canales a asignar.


controlBus: Asignar un índice de canal de control en el servidor.

  • channels: Número de canales a asignar.


buffer: Si no se especifican, asigna un número de buffer y reserva la memoria en el servidor.

  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 
  • numBufs: Número de números contiguos de buffer a reservar (predeterminado = 1).
  • numFrames: Número de cuadros.
  • numChannels: Número de canales.


allocRead: Leer un archvo de disco a la memoria del servidor. El archivo es cerrado cuando acaba.

  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 
  • path: Ruta al archivo en el disco.
  • firstFileFrame: Dónde empieza a leer el archivo.
  • numFrames: Número de cuadros. So no es dado, el archivo completo es leído.


cue: Adelantar la carga de un archivo de sonido (para usarse con DiskIn).

  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 
  • path: Ruta al archivo en el disco.
  • firstFileFrame: Dónde empieza a leer el archivo.
  • numFrames: Número de cuadros. So no es dado, el archivo completo es leído.
  • firstBufferFrame: En qué cuadro empezar a leer los datos del archivo.
  • leaveOpen: 1 = dejar el archivo abierto (para uso con DiskIn). 0 = cerrar el archivo después de leerse. Por defecto es  0.


table: Llenar un  buffer con datos preestablecidos. Esto usa  /b_setn para transferir los datos, de manera en que todos los datos quepan en un datagrama. Tomará algo de experimentación encontrar el límite superior.  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 

  • amps: Los valores a poner en el buffer. Deben de ser Floats (número de punto flotante).


cheby: Genera una función de transferencia Chebyshev para moldeado de onda (waveshaping).

  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 
  • numFrames: Número de cuadros, deberá ser una potencia de 2.
  • numChannels: Número de canales.
  • genflags: Tres señales, asociadas con números: normalizar = 1, comoTablaDeOnda = 2, limpiarPrimero = 4. Sumar los número para la señal deseada. Normalmente todas las señales son ciertas, sumando 7. 
  • amps: La amplitud de cada parcial (es decir el coeficiente polinomial).


sine1: Imita el método sine1 para Buffer, generando una tabla de onda con espectro armónico de múltiples enteros usando las amplitudes de parciales dados. 

  • bufNum: (Opcional) Número de buffer. Si no es dado, un número libre será obtenido del servidor. 
  • numFrames: Número de cuadros, deberá ser una potencia de 2.
  • numChannels: Número de canales.
  • genflags: Ver arriba.
  • amps: Arreglo de amplitudes para cada parcial.


sine2: Igual que sine1, pero la proporción de frecuencias para cada parcial también es dada. 

  • Mismo  argumentos que con sine1, más:
  • freqs: Arreglo de frecuencias para cada parcial. 1.0 es la frecuencia fundamental; su sinusoide ocupa la duración entera del buffer.


sine3: Igual que sine2, pero la fase de cada parcial también puede ser suministrada.

  • Mismos argumentos que con sine1, más:
  • phases: Arreglo de fases para cada parcial, dados en radianes (0.0 - 2pi).




Salida MIDI


midi: Manda uno de varios tipos de mensajes MIDI a un objeto MIDIOut.

  • Argumentos estándares para el control de Temporalización y de Nodos (excepto timingOffset, que no es usado)
  • midicmd: El tipo de mensaje MIDI a mandar. Esto también determina otros argumentos que deben estar presentes en el evento. 
  • midiout: El objeto MIDIOut, que conecta con uno de los aparatos MIDI enlistados en MIDIClient.destinations.
  • chan: El número de canal MIDI (0-15) en el aparato que debe recibir el mensaje. Esto aplica a todos los comandos midi (midicmd) excepto los globales (smpte, songPtr, sysex).


Comandos midi disponibles


  • noteOn: Comienza una nota y opcionalmente la detiene. Si se dan múltiples frecuencias, un par noteOn/noteOff es mandado para cada una y \strum es también admitido. 
    • chan: Canal MIDI  (0-15).
    • midinote: Número de nota a disparar. Esto se puede calcular a partir de la jerarquía de alturas descrita en GP_07_Conversiones_de_Valor (con la excepción de que sólo se admite temperamento igual de 12 notas por octava). 
    • amp: Velocidad MIDI = amp / 12.
    • sustain: Cuántos pulsos esperar antes de enviar el correspondiente mensaje de noteOff. Si no es dado directamente, es calculado como ~sustain = ~dur * ~legato * ~stretch (al igual que en el evento tipo \note).
    • hasGate: Normalmente cierto. Si es falso, el mensaje de noteOff no será enviado.


  • noteOff: Mandar un mensaje de noteOff explícito (útil si hasGate es falso en el evento noteOn).
    • chan: Canal MIDI (0-15).
    • midinote: Número de nota.
    • amp: Velocidad de liberación (admitido por algunos sintetizadores). 


  • allNotesOff: Mensaje de "Pánico", mata todas las notas en un canal. 
    • chan: Canal MIDI (0-15).


  • control: Mensaje de controlador continuo (continuous controller).
    • chan: Canal MIDI (0-15).
    • ctlNum: Número de controlador para recibir el nuevo valor. 
    • control: Nuevo valor (0-127).


  • bend: Mensaje de pitch bend ('doblar' altura).
    • chan: Canal MIDI (0-15).
    • val: Nuevo valor (0-16383). 8191 es en el centro.


  • touch: Mensaje de aftertouch ('toque posterior').
    • chan: Canal MIDI (0-15).
    • val: Nuevo valor (0-127).


  • polyTouch: Mensaje de aftertouch polifónico (no admitido por todos los sintetizadores).
    • chan: Canal MIDI (0-15).
    • midinote: Número de nota que obtendrá el nuevo valor de toque. Como en noteOn, puede ser calculado a partir de la jerarquía estándar de alturas. 
    • polyTouch: Nuevo valor (0-127).


  • program: Mensaje de cambio de programa.
    • chan: Canal MIDI (0-15).
    • progNum: Número de programa (0-127).


  • smpte: Mandar mensajes de MIDI Time Code (Código de Tiempo MIDI).
    • Argumentos: cuadros, segundos, minutos, horas, velocidad de cuadros. 


  • songPtr: Mensaje de indicador de canción (song pointer).
    • songPtr: Valor del indicador (0-16383).


  • sysex: Mensajes de sistema exclusivo.
    • array: Un arreglo tipo Int8Array con los bytes de sysex bytes en orden. Muy importante: Normalmente los arreglos se expanden en multicanales en los patrones. Se debe envolver el arreglo Int8Array dentro de otro arreglo para prevenir esto. Escribir  [Int8Array[...]], no solamente Int8Array[...].



Misceláneos


phrase: Ver recursive_phrasing (JITLib, fraseo recursivo).


setProperties: Ajustar variables que pertenecen a un Objeto dado. Un posible uso es controlar un GUI (Interface Gráfico de Usuario) usando un patrón.

receiver: El objeto a ser modificado.

  • args: La lista de nombres de variables a ajustar en el receptor. El receptor deberá contar con métodos de ajustar -- nombreDeVariable_ -- para cada uno de estos. Nuevos valores serán buscados en el evento. 


// Visualizar movimiento Browniano

w = Window("Movimiento Browniano", Rect(10, 100, 500, 50));

x = Slider(w, Rect(10, 15, 480, 20));

w.front;


p = Pbind(

\type, \setProperties,

\receiver, x,

// esto significa llamar x.value_() en cada evento

\args, [\value],

// y buscar el valor bajo la llave \value

\value, Pbrown(0, 1, 0.1, inf),

\delta, 0.1

).play;


p.stop;
