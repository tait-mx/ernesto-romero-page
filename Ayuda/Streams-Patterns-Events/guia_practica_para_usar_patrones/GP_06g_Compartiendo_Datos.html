Compartiendo datos entre patrones


Hasta ahora hemos visto patrones que son independientes entre sí. Cada Pbind trabaja sobre su propia información, que no es accesible a otros Pbinds. También, por ejemplo, el patrón 'degree' ('grado') en un Pbind no está al tanto de lo que el patrón 'dur' está haciendo. Hacer que estos datos sean accesibles añade a la inteligencia musical de los patrones.


Hay varias maneras distintas de transmitir información de un patrón a otro. La primera técnica, más sencilla, es leer valores del evento que actualmente está siendo procesado. La segunda es pasar información de un patrón de eventos a otro. Ya que ambos son patrones de eventos, sus eventos resultantes son diferentes y por lo tanto la primera técnica no es aplicable.  


Leyendo valores del evento actual


Dentro de un Pbind los patrones de valores pueden leer otros valores que ya hayan sido colocados en el evento. El patrón Pkey busca la llave en el evento que actualmente está siendo procesado y retorna su valor. De ahí, uno puede realizar cualquier otra operación de patrones con ese valor:  patrones de filtro, operadores matemáticos, etc. 


  • Pkey(llave): Lee la 'llave' en el evento de entrada. Retorna valores hasta que el evento de entrada deje de producir valores para esa llave (es decir, que la llave retorne nulo). No hay argumento 'repeticiones'. Si necesita limitar el número de valores, envuelva Pkey en un Pfin.


p = Pkey(\a).asStream;


// El valor de entrada es un evento con a = 2, \b = 3; 'next' ('siguiente') resultado es 2

p.next((a: 2, b: 3));


// También podemos hacer cálculos con el evento de entrada

p = (Pkey(\a) * Pkey(\b)).asStream;

p.next((a: 2, b: 3)); // retorna 6 == 2 * 3



En este ejemplo, staccato vs. legato es calculado basado en el grado de la escala: las notas graves son más largas y las agudas son más cortas. ¡Esto apenas rasca en la superficie de esta técnica!


Hay que tener en cuenta que Pkey sólo puede ver hacia atrás llaves que ya fueron declaradas anteriormente en la definición del Pbind. Pbind procesa las llaves en el orden dado. En el ejemplo, no funcionaría poner primero 'legato', ya que se refiriere a 'degree' que viene después, porque el valor para 'degree' todavía no se ha producido. 


// algo simple - mientras más alta la nota, más corta la articulación

(

p = Pbind(

\degree, Pseq([Pseries(-7, 1, 14), Pseries(7, -1, 14)], inf),

\dur, 0.25,

// \degree se encuentra ANTES en el Pbind

\legato, Pkey(\degree).linexp(-7, 7, 2.0, 0.05)

).play;

)


p.stop;



Otros patrones para guardar información


Estos patrones representan tres estrategias diferentes para hacer que la información persista. Estos patrones representan tres distintas estrategias para persistir información de un patrón y hacerla accesible a otros. 


Penvir(ambiente, patrón, independiente): Los Flujos que evalúan patrones son usualmente Rutinas, y las Rutinas tienen la característica especial de acordarse del Ambiente (Environment) que estaba en vigor la última vez que cedió (yield). Las Rutinas se encargan de restaurar ese Ambiente la siguiente vez que se  despierta.  Penvir establece un ambiente dentro del cual 'patrón' correrá. El ambiente puede ser inicializado con valores o puede en un principio estar vacío y poblarse por elementos del patrón. El ambiente está separado del evento procesado (y de hecho, el patrón puede ser tanto de valores como de eventos). El acceso al ambiente se realiza a partir de patrones de funciones como Pfunc, Pfuncn, Proutine, .collect, .select, .reject y similares. 


La señal 'independiente' especifica si el ambiente se mantiene o no separado de cada flujo hecho a partir del Penvir. Si cierto (el predeterminado), cuando el Penvir sea incrustado en un flujo, un nuevo ambiente es creado que hereda los valores iniciales proveídos en 'ambiente'. Si es falso, el mismo ambiente es usado para cada flujo. En ese caso, el mismo ambiente podría también ser usado en distintos patrones Penvir, y las modificaciones del ambiente por un Penvir se reflejaría en todos los demás -- de ahí su utilidad para compartir datos. 


Pfset(función, patrón, funciónDeLimpieza): Cuando es incrustado, Pfset crea un ambiente y lo puebla usando asignaciones de variables de ambiente contenidas dentro de la 'función' que se provee. Para cada llamada al 'siguiente', los valores en el ambiente predeterminado son insertadas en el prototipo de evento antes de que el patrón vástago se evalúe. Esta es una manera de ajustar valores predeterminados para un patrón. Puede ser usado también para cargar objetos en el servidor, aunque esto toma algo de cuidado ya que esos objetos volverían a ser cargados cada vez que Pfset sea reproducido. Uno es responsable de liberar los objetos creados de esta manera mediante la funciónDeLimpieza. (Otra manera de hacerlo es con Pproto; ver GP_06f_Control_del_servidor)



(

SynthDef(\playbuf, { |bufnum, start, dur = 1, amp = 0.2, out|

var sig = PlayBuf.ar(1, bufnum, BufRateScale.ir(bufnum), 0, start);

sig = sig * amp * EnvGen.kr(Env.linen(0.01, dur, 0.01), doneAction: 2);

Out.ar(out, sig ! 2)

}).memStore;

)


(

TempoClock.default.tempo = 1;

p = Pfset({

~buf = Buffer.read(s, "sounds/a11wlk01.wav");

0.2.yield; // sync parece ser incompatible con patrones

~bufFrames = ~buf.numFrames;

}, Pbind(

\instrument, \playbuf,

// el acceso a los recursos del protoevento es mediante Pkey

\bufnum, Pkey(\buf),

\dur, Pwhite(1, 4, inf) * 0.25,

// el límite supero del Pwhite está basado en la duración para 

// que start + (dur * samplerate) nunca llega más allá de la longitud del buffer

\start, Pwhite(0, Pkey(\bufFrames) - (Pkey(\dur) * 44100), inf)

), { defer(inEnvir { "liberando buffer".postln; ~buf.free }, 1.1) }).play;

)


// muestra un número buffer asignado ('true' ContiguousBlock)

s.bufferAllocator.debug;


p.stop;



Plambda(patrón, ambienteDeEvento): Mantiene un ambiente cuyo alcance es el evento actual mientras es procesado. Los valores pueden asignarse al evento usando Plet(llave, patrón, retorno) y leerse usando Pget(llave, predeterminado, repeticiones). Pget es similar a Pkey pero tiene el argumento 'repeticiones' controlando el número de valores retornados así como el argumento 'predeterminado' que se usará si la llave dada no es encontrada dentro del evento. 


Una característica única de Plambda/Plet/Pget es su capacidad de asignar un valor en su ambiente de evento y simultáneamente retornar otro valor en el ambiente principal. Plet asigna el valor de su 'patrón' dentro del alcance del ambiente de evento. El argumento 'retorno' es opcional; si se provee, le regresa el valor de retorno al Pbind. 


Plambda remueve el ambienteDeEvento antes de retornar el evento final a quien lo llamó. Uno puede ver lo que ocurre dentro del alcance del ambiente mediante un .trace ('segur la huella/pista') en el patrón interno.  


p = Plambda(

Pbind(

\a, Plet(\z, Pseries(0, 1, inf), Pseries(100, -1, inf)),

\b, Pget(\z, 0, inf) * 2

).trace(key: \eventScope, prefix: "\nscope: ")

).asStream;


p.next(());


Algo similar se puede hacer con Pkey, usando valores intermedios en el evento que no correspondan a  nombres de controles de un SynthDef. No hace ningún daño tener valores extra en un evento que el sinte no utilice; solo los que se requieren se envían al servidor. A menudo esto es más simple que Plambda, pero habrán casos donde la única manera de hacerlo sea con Plambda. 


p = Pbind(

\z, Pseries(0, 1, inf),

\a, Pseries(100, -1, inf),

\b, Pkey(\z) * 2

).asStream;


p.nextN(5, ()).do(_.postln);



Comunicando valores entre patrones de eventos separados


Pasar valore de un Pbind a otro requiere varios pequeños trucos. Primero se guardan eventos completados en un lugar accesible. Ningún patrón ni el ReproductorDePatronesDeEvento (EventStreamPlayer) salvan los eventos terminados; sin embargo, llamando a 'collect' en un patrón adjunta una acción hecha a la medida a ejecutarse para cada evento resultante. Aquí salvamos el evento a una variable de ambiente, pero podría también ir a la librería global, una variable declarada o cualquier otra estructura de datos. 


Segundo, tenemos que asegurar que el patrón fuente sea evaluado antes de cualquier patrón cliente que dependa de los valores de la fuente. La única manera de hacer esto es planificando a la fuente ligeramente antes, ya que los elementos que se planifican al mismo tiempo en un reloj pueden llegar a ejecutarse en cualquier orden. (No hay un mecanismo de prioritización para hacer que un proceso corra antes que otro). No obstante, este mecanismo de planificación no deberá afectar la precisión temporal del audio. 


La solución consiste en un mecanismo de desplazamiento en la secuenciación de eventos que retrasa el sonido de un evento por un cierto número de pulsos. En el ejemplo, el patrón del bajo se planifica 0.1 pulsos antes de los pulsos que caen en números enteros (mientras que el patrón de acordes corre exactamente en pulsos enteros). El patrón del bajo opera con un desplazamiento temporal de 0.1, retrasando el sonido para que suceda en pulsos enteros. Ambos patrones suenan juntos en el servidor a pesar de que su secuenciación temporal es diferente en el cliente. 



Pulso

Tiempo en el cliente 

Tiempo en el servidor

0.9

Se calcula el evento del bajo

(el evento del bajo es retrasado por 0.1, nada ocurre aquí)

1.0

Se calcula el evento del acorde

Tanto bajo como acorde producen un sonido


(

TempoClock.default.tempo = 1;


~bajo = Pbind(

\degree, Pwhite(0, 7, inf),

\octave, 3, // dos octavas abajo

\dur, Pwhite(1, 4, inf),

\legato, 1,

\amp, 0.2

).collect({ |event|

~ultimoEventoDelBajo = event;

// aquí se determina el retraso de su reproducción en el servidor:

}).play(quant: Quant(quant: 1, timingOffset: 0.1));


// forma corta para el objeto Quant: #[1, 0, 0.1]


~acordes = Pbind(

\notaSuperior, Pseries(7, Prand(#[-2, -1, 1, 2], inf), inf).fold(2, 14),

\notasTriadaDelBajo, Pfunc { ~ultimoEventoDelBajo[\degree] } + #[0, 2, 4],

// fusionar la triada hacia la notaSuperior

// sube las notas de la triada a la octava siguiente a la notaSuperior

// div: es división entera, por lo que x div: 7 * 7 significa el siguiente

// múltiplo abajo del 7

\fusionar, 

(Pkey(\notaSuperior) - Pkey(\notasTriadaDelBajo)) div: 7 * 7 + Pkey(\notasTriadaDelBajo),

// añadir notaSuperor al arreglo, si no es que ya estaba ahí

\degree, Pfunc { |ev|

if(ev[\fusionar].detect({ |item| item == ev[\notaSuperior] }).isNil) {

ev[\fusionar] ++ ev[\notaSuperior]

} {

ev[\fusionar]

}

},

\dur, Pwrand([Pseq([0.5, Pwhite(1, 3, 1), 0.5], 1), 1, 2, 3], #[1, 3, 2, 2].normalizeSum, inf),

\amp, 0.05

).play(quant: 1);

)


~bajo.stop;

~acordes.stop;


El patrón de acordes demuestra algunas maneras en las cuales se puede expresar una lógica de alto nivel mediante patrones. El objetivo es trasponer las notas de la triada en posición fundamental por encima de las notas del bajo, de manera en que todas las notas caigan dentro de la octava inferior a una nota superior (que se escoge mediante movimiento por grados). Pkey(\notaSuperior) - Pkey(\notasTriadaBajo) da el número de pasos de transposición necesarias para mover las notas de la triada hacia la nota superior; luego los pasos de transposición son truncados a la siguiente octava inferior (x div: 7 es división de enteros y produce un número de octava; multiplicando por 7 da el número de grados de la escala para esa octava). 


f = { |notaSuperior, triada|

var x;

x = (notaSuperior - triada).debug("pasos de transposición inicial");

x = (x div: 7).debug("octavas a trasponer");

x = (x * 7).debug("pasos a transponer");

x + triada

};


f.value(7, #[0, 2, 4]);

--> [ 7, 2, 4 ] (triada en primera inversión)


Entonces se verifica el arreglo transpuesto para ver si la nota superior está incluida en él. Si no lo está, es añadida para que la melodía siempre esté presente.


Nótese que las operaciones perezosas en los patrones definen la mayor parte de este comportamiento; sólo la verificación condicional del arreglo tuvo que ser escrito como función. 


El anterior ejemplo rompe con uno de los principios del diseño de patrones. Idealmente, debería ser posible reproducir un solo objeto de patrón muchas veces simultáneamente sin que los distintos flujos interfirieran entre sí. El hecho de salvar la nota del bajo en una variable de ambiente significa que flujos concurrentes no trabajarían juntos ya que no pueden usar la misma variable de ambiente al mismo tiempo. La aproximación anterior, sin embargo, permite que los dos patrones se detengan y arranquen independientemente y en cualquier momento se puede añadir nuevos patrones que dependan del bajo. En algunas circunstancias musicales este tipo de flexibilidad es más importante que el respetar el ideal de diseño de patrones. 


Es posible, usando Ptpar y Penvir crear ambientes independientes para el almacenaje de eventos como parte del patrón mismo. De manera predeterminada, Penvir crea una nueva copia de su ambiente para cada flujo, garantizando su independencia. Mientras el patrón está corriendo, ~ultimoEventoDelBajo = event salva el evento en la copia del ambiente de almacenaje que pertenece al flujo y es accesible a ambos Pbinds porque los dos están bajo control del Penvir (indirectamente a través del Ptpar).


(

p = Penvir((), Ptpar([

0.0, Pbind(

\degree, Pwhite(0, 7, inf),

\octave, 3, // 2 octavas abajo

\dur, Pwhite(1, 4, inf),

\legato, 1,

\amp, 0.2,

// esta vez aquí se determina el retraso en su reproducción:

\timingOffset, 0.1 

).collect({ |event|

~ultimoEventoDelBajo = event;

}),

0.1, Pbind(

\notaSuperior, Pseries(7, Prand(#[-2, -1, 1, 2], inf), inf).fold(2, 14),

\notasTriadaDelBajo, Pfunc { ~ultimoEventoDelBajo[\degree] } + #[0, 2, 4],

\fusionar, (Pkey(\notaSuperior) - Pkey(\notasTriadaDelBajo)) 

div: 7 * 7 + Pkey(\notasTriadaDelBajo),

\degree, Pfunc { |ev|

if(ev[\fusionar].detect({ |item| item == ev[\notaSuperior] }).isNil) {

ev[\fusionar] ++ ev[\notaSuperior]

} {

ev[\fusionar]

}

},

\dur, Pwrand([Pseq([0.5, Pwhite(1, 3, 1), 0.5], 1), 1, 2, 3], #[1, 3, 2, 2].normalizeSum, inf),

\amp, 0.05

)

])).play;

)


p.stop;
