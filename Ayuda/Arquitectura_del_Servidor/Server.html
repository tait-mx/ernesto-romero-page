Server objeto que representa a una aplicación sc-server


superclase: Model


Un objeto Server es la representación del lado del cliente de una aplicación servidor y es usado para controlar a esta aplicación desde la aplicación SuperCollider lenguaje. (Ver ClienteVsServidor para más detalles sobre esta distinción). La clase Server envía mensajes OSC y tiene un número de asignadores que llevan el control de IDs para nodos, canales y buffers. La aplicación servidor es un programa de línea de comandos, por lo tanto, todos los comandos, aparte de los mensajes OSC, son comandos Unix. La aplicación servidor representada por un objeto Server puede estar corriendo en la misma máquina que el cliente (en el mismo espacio de direcciones que la aplicación lenguaje o por separado; ver más abajo) o en una máquina remota.


La mayor parte de las opciones de una instancia de la clase Server se controlan a través de su instancia de la clase ServerOptions. Ve el archivo de ayuda de ServerOptions para más detalles.


Rutas


Las aplicaciones servidor que corren en la máquina local tienen dos variables de entorno Unix: SC_SYNTHDEF_PATH y SC_PLUGIN_PATH. Estas indican las carpetas de SynthDefs y de plugins de UGens que serán cargadas al arrancar. Estas carpetas pueden ser definidas dentro de SuperCollider, utilizando los métodos getenv y setenv de la clase String, y serán cargadas además de las carpetas por defecto de SynthDefs y de plugins, las cuales están codificadas "en duro".


// todas las definiciones en este directorio serán cargadas cuando el servidor local arranque

"SC_SYNTHDEF_PATH".setenv("~/scwork/".standardizePath); 

"echo $SC_SYNTHDEF_PATH".unixCmd;


El grupo por defecto


Cuando un servidor es arrancado, existe un grupo de nivel superior, con un ID de 0, el cual define la raíz del árbol de nodos (este grupo está representado por la subclase RootNode de la clase Group). Si la aplicación servidor es arrancada desde SCLang (en oposición a desde la línea de comandos), el método initTree será llamado automáticamente después de arrancar. Esto también creará un grupo por defecto (default_group), con un ID de 1, el cual es el grupo por defecto de todos los nodos en el estilo objeto. Esto provee un árbol de nodos básico y predecible para que métodos como Server:scope, Server:record, etc. puedan funcionar si tener problemas de orden de ejecución. El grupo por defecto es persistente, i.e. es recreado después de reiniciar, presionar Command-., etc. Ver RootNode y default_group para más información. Toma nota de que si un servidor ha sido arrancando desde la línea de comandos, deberás llamar a initTree manualmente para que inicialice al grupo por defecto, si es que lo necesitas. Ver initTree más abajo.


Local vs. Internal


En general, al trabajar con una sola máquina, probablemente utilicemos uno de los dos objetos Server que son creados al arrancar el programa y guardados en las variables de clase local e internal. Por defecto, dos ventanas de gráfica de usuario son creadas para controlarlos. La diferencia entre los dos es que el servidor local corre como una aplicación separada, con su propio espacio de direcciones, mientras que el servidor interno corre dentro del mismo espacio que la aplicación lenguaje/cliente. El servidor interno tiene la ventaja de poder accesar memoria compartida, permitiendo de esta manera conveniencias como osciloscopios (ver más abajo) y SharedIn/SharedOut. El servidor interno también minimiza la latencia en el envío de mensajes. El servidor local, y cualquier otra aplicación servidor que corra en tu máquina local, tiene la ventaja de que si el lenguaje se "crashea" (y con esto posiblemente tu pieza), el servidor continuará corriendo. De esta forma, utilizar al servidor local es una disposición inherentemente más robusta.


El servidor por defecto


Siempre hay un servidor por defecto, el cual está guardado en la variable de clase default. Cualquier instancia de la clase Synth o Group que sea creada sin un objetivo ("target") será creada en el servidor por defecto. Al arrancar, el servidor local (ver más arriba) es definido como servidor por defecto, pero cualquier otro servidor puede serlo.



Métodos de Clase


*new(name, addr, options, clientID)

name - un símbolo; cada objeto Server es guardado con su nombre en una variable de clase global.

addr - una instancia de NetAddr opcional, que provea la dirección IP del ordenador anfitrión y el número puerto. El valor por defecto es la dirección del localhost utilizando el puerto 57110; igual que el servidor local.

options - una instancia de ServerOptions opcional. Si nil, una instancia de ServerOptions será creada, utilizando los valores por defecto.

clientID - un entero. En situaciones multi-cliente, a cada cliente se le puede dar un rango seprado de nodeIDs. El valor por defecto es 0.



*local - devuelve al servidor local, guardado en la variable de clase local (ya creada en initClass)


*internal - devuelve al servidor interno, guardado en la variable de clase internal (ya creada en initClass)


*default - devuelve el servidor por defecto. Por defecto, éste es el servidor local (ver arriba)


*default_(aServer) - define a aServer como servidor por defecto


Server.default = Server.internal; // define al servidorinterno como servidor por defecto

*all - el conjunto de todos los servidores


*quitAll - apaga todos los servidores registrados


*killAll - consulta al sistema acerca de las aplicaciones sc-server y las apaga a la fuerza

*freeAll - libera a todos los nodos en todos los servidores registrados



Métodos de Instancia

sendMsg(arg1, arg2, arg3, ... argN) - manda un mensaje OSC al servidor.


s.sendMsg("/s_new", "default", s.nextNodeID, 0, 1);


sendBundle(time, array1, array1, array1, ... arrayN) - manda un paquete OSC al servidor. Ya que la red puede tener un rendimiento irregular, el argumento time permite que el paquete sea evaluado en un punto específico en el futuro. Por lo tanto, todos los mensajes son síncronos en relación a sí mismos, pero retrasados una misma cantidad de tiempo. Si un paquete llega tarde, el servidor contesta con un mensaje de "tarde", pero lo evalúa de todas formas.

s.sendBundle(0.2, ["/s_new", "default", x = s.nextNodeID, 0, 1], ["/n_set", x, "freq", 500]);

sendRaw(aRawArray)


listSendMsg([arg1, arg2, arg3, ... argN]) - como sendMsg, pero toma un arreglo como argumento.


listSendBundle(time, [array1, array1, array1, ... arrayN]) - como sendBundle, pero toma un arreglo como argumento. Esto te permite reunir mensajes en un arreglo y mandarlos después.


s.listSendBundle(0.2, [["/s_new", "default", x = s.nextNodeID, 0, 1], 

["/n_set", x, "freq", 600]]);

sendSynthDef(name, dir) - envía al servidor un SynthDef que fue escrito en un directorio local

loadSynthDef(name, completionMsg, dir) - carga un SynthDef que reside en un directorio remoto


loadDirectory(dir, completionMsg) - carga todos los SynthDefs en el directorio dir. El argumento dir es una Cadena de Caracteres ("String") con una ruta válida.

nextNodeID - obtén un ID de nodo único


nextPermNodeID - obtén un ID de nodo permanente. Este ID de nodo se encuentra en un rango reservado y se mantendrá hasta que lo liberes explicitamente


freePermNodeID - libera un ID de nodo permanente para que sea reutilizado después

wait(responseName) - este mensaje puede ser utilizado desde dentro de una Routine para esperar una respuesta del servidor


waitForBoot(func, limit) - evalúa la función func tan pronto como el servidor haya arrancando. Si está corriendo, es evaluada inmediatamente. Si no está corriendo, arranca al servidor y evalúa la función. El argumento limit indica el máximo número de intentos. (5 veces/segundo)


doWhenBooted(func, limit) - evalúa la función tan pronto como el servidor haya arrancando. Si está corriendo, es evaluada inmediatamente. El argumento limit indica el máximo número de intentos. (5 veces/segundo)


boot(startAliveThread) - arranca el servidor remoto, crea nuevos asignadores. startAliveThread: si es definido como falso, el servidor no es consultado para dar información a la ventana. N.B. No puedes arrancar localmente una aplicación servidor que se encuentre en una máquina remota.


quit - apaga la aplicación servidor


reboot - apaga y reinicia la aplicación servidor


freeAll - libera todos los nodos en este servidor


status - consulta el estado del servidor


notify(flag) - el servidor manda notificaciones, por ejemplo: si un nodo fue creado, un mensaje 'tr' de un SendTrig o una acción /done. Si el argumento flag es definido como falso, estos mensajes no son enviados. El valor por defecto es true (cierto).


dumpOSC(code)

code:

0 - apaga el vaciado ("dumping")

1 - imprime el análisis sintáctico del contenido del mensaje

2 - imprime el contenido en hexadecimal

3 - imprime ambas representanciones del contenido: la analizada y la hexadecimal

queryAllNodes - Imprime en la ventana "post" una representación del árbol de nodos actual de este servidor. Muy útil durante la depuración de programas. Utiliza g_dumpTree en los servidores locales y  g_queryTree en los remotos. Ve Group y Server-Command-Reference para más información.

s.boot;

s.queryAllNodes; // observa el nodo raíz o "root node" (ID 0) y el 

// grupo por defecto o "default group" (ID 1)

s.quit;

ping(numberOfTimes, waitBewteen, completionFunction)

mide el tiempo entre servidor y cliente, el cual puede variar. La función en el argumento 

completionFunction es evaluada después del número de mediciones especificado

en el argumento numberOfTimes; el máximo resultante es enviado a la función como

argumento.


options - devuelve al objeto ServerOptions de este servidor. Los cambios entrarán en efecto cuando el servidor sea reiniciado.


options_(aServerOptions) - define el objeto ServerOptions de este servidor. Los cambios entrarán en efecto cuando el servidor sea reiniciado.


defaultGroup - devuelve al grupo por defecto de este servidor.

volume - una instancia de la clase Volume que corre después del grupo por defecto.


volume_(level) - ajusta el volumen de la salida del servidor a través de su instancia de la clase Volume. Los niveles están en decibelios.


mute - silencia la salida del servidor. Esto puede ser conmutado desde la ventana del servidor, con la letra 'm'.


unmute - deja de silenciar al servidor. Esto también puede ser conmutado desde la ventana del servidor, con la letra 'm'.


Empaquetamiento Automático de Mensajes


La clase Server provee soporte para el empaquetamiento automático de mensajes. Esto asegura la ejecución sincrónica y es bastante conveniente en 'estilo objeto'. Ver también bundledCommands.


makeBundle(time, func, bundle) - La función func es evaluada, y todos los mensajes OSC generados por ella son diferidos y añadidos al paquete. Este método devuelve al paquete para que, de ser necesario, pueda continuar utilizándose. Si el argumento time es definido como nil o como un número, el paquete va a ser enviado automáticamente y ejecutado después del correspondiente retraso en segundos. Si time es definido como falso, el paquete no será enviado. El argumento bundle te permite pasar un paquete preexistente y continuar añadiéndole mensajes. Si sucede une error al evaluar func, este método lanzará un error y detendrá el aplazamiento de los mensajes.


s.boot;

(

// envía un sintetizador al servidor

SynthDef("tpulso", { arg out=0,frec=700, serruchofrec=440.0;

Out.ar(out, SyncSaw.ar(frec, serruchofrec, 0.1) )

}).send(s);

)

// todos los comandos OSC generados en la función contenida en el siguiente código

// van a ser añadidos a un paquete y ejecutados simultáneamente después de 2 segundos

(

s.makeBundle(2.0, {

x = Synth.new("tpulso");

a = Bus.control.set(440);

x.map(\frec, a);

});

)

x.free;

// no envíes

(

b = s.makeBundle(false, { 

x = { PinkNoise.ar(0.1) * In.kr(0, 1); }.play;

});

)

// ahora pasa b como un paquete preexistente y arranca ambos sintetizadores sincrónicamente

(

s.makeBundle(nil, { // nil ejecuta tan pronto como sea posible

y = { SinOsc.kr(0.2).abs }.play(x, 0, 0, \addBefore); // sine envelope

}, b);

)

x.free; y.free;

// lanza un error

(

try {

s.makeBundle(nil, {

s.farkermartin;

});

} { |error|

("Mira Ma, las operaciones normales se reaunudan a pesar de:\n" + error.errorString).postln; 

x = { FSinOsc.ar(440, 0, 0.2) }.play; // esto funciona bien

}

)

x.free;


bind(func) - Igual que en makeBundle, la función func es evaluada, y todos los mensajes OSC generados por ella son diferidos y añadidos a un paquete, el cual es enviado al servidor, utilizando la latencia por defecto de éste.


// ejemplo

(

s.bind {

a = { |frec=100| SinOsc.ar(frec, LFTri.ar(frec)) * 0.2 }.play;

s.sync;

a.set(\frec, 400);

}

)


Controles Compartidos


El servidor interno tiene un número de canales de control compartidos. Sus valores pueden ser ajustados o consultados utilizando los siguientes métodos:


getSharedControl(num) - obtén el valor actual de un canal de control compartido. El argumento num es el índice del canal a consultar. Este comando es sincrónico y sólo funciona con el servidor interno.


setSharedControl(num, value) - define el valor de un canal de control compartido. El argumento num es el índice del canal a definir. Este comando es sincrónico y sólo funciona con el servidor interno.


allocSharedControls(numControls) - define el número de canales de control compartidos. Esto debe de ser hecho antes de que el servidor interno sea arrancado. El número por defecto es 1024.


Arboles de Nodos Persistentes


La clase ServerTree puede ser utilizada para guardar funciones, las cuales serán evaluadas después de que el servidor sea arrancado, después de que todos los nodos sean liberados, y después de que Command-. sea presionado. Esto permite, por ejemplo, crear una estructura de nodos básica que sea persistente. ServerTree es evaluado en el método initTree, después de que el grupo por defecto es creado, por lo que se puede contar con su existencia.


initTree - Este método inicializa al grupo por defecto (default_group) y corre a ServerTree. Este método es llamado automáticamente cuando inicias un servidor desde el lenguaje. N.B. Si prendiste una aplicación servidor desde la línea de comandos y necesitas de esta funcionalidad, tendrás que llamar a initTree manualmente.


s.quit;

f = {Group.new(s.defaultGroup); "Tambien puede evaluarse otro codigo".postln;};

ServerTree.add(f);

s.boot;

s.queryAllNodes; // observa al grupo dentro del grupo por defecto

ServerTree.remove(f);


ServerBoot y ServerQuit proveen una funcionalidad similar al arrancar y al apagar el servidor.

Osciloscopio


Sólo funciona con el servidor interno y, actualmente, sólo en OS X.


Ver Stethoscope para más detalles.


scope(numChannels, index, bufsize, zoom, rate) - Abre una ventana de osciloscopio que muestra la señal de salida del servidor. 

numChannels - el número de canales que mostrar. El valor por defecto es 2.

index - el primer canal que mostrar. El valor por defecto es 0.

bufsize - el tamaño del buffer para el ScopeView. El valor por defecto es 4096.

zoom - un valor de zoom para el eje de la X del osciloscopio. Los valores grandes muestran más. El valor por defecto es 1.

rate - si mostrar canales de audio o de control (ya sea \audio o \control)


Soporte para Grabaciones


Los siguientes métodos son para usarse por conveniencia. Si necesitas hacer grabaciones con tiempos de inicio y de fin con una precisión a nivel de muestra de digital, deberás crear tus propios nodos. Ve el archivo de ayuda de DiskOut para más información. Para grabaciones a tiempo diferido, ve el archivo de ayuda de Non-Realtime-Synthesis. 


Esta funcionalidad también está disponible a través del botón de grabación de las ventanas de los servidores. Presionándolo una vez se llama a prepareForRecord, presionándolo otra vez se llama a record, y presionándolo una tercera vez se llama a stopRecording (ver más abajo). Al hacer esto, el archivo creado estará en tu carpeta de grabaciones y tendrá como nombre la fecha y la hora del momento en que se haya llevado a cabo la grabación. La ubicación por defecto de la carpeta de grabaciones varía de una plataforma a otra pero siempre es guardada en thisProcess.platform.recordingsDir. Definir a esta variable te permite cambiar la ubicación por defecto.


NOTA: record coloca al sintetizador que graba después del grupo por defecto del servidor y este sintetizador utiliza In.ar. Es por esto que si añades nodos después del sintetizador que lleva a cabo la grabación, su salida no será capturada. Para prevenir esto, utiliza, ya sea objetos Node (los cuales usan el grupo por defecto como objetivo) o (al utilizar "estilo envío de mensajes") usa como objetivo un ID de nodo de 1.


s.sendMsg("/s_new", "default", s.nextNodeID, 1,1);


Para más detalles sobre este tema ve Order-of-execution, default_group, y NodeMessaging.


prepareForRecord(path) - Asigna el buffer necesario, etc. para grabar la salida del servidor. (Ve record más adelante) El argumento path es una Cadena de Caracteres ("String") que representa la ruta y el nombre del archivo generado. Si no especificas una ruta,  entonces el archivo va a ser creado en tu carpeta de grabaciones (ver la nota anterior al respecto) con el nombre SC_estaFecha_y_Hora. Los cambios al tipo de encabezado o al formato de muestra o al número de canales, deben hacerse ANTES de llamar a este método.

record - Inicia o reanuda la grabación de la salida. Tienes que haber llamado antes al método prepareForRecord (ver arriba).

pauseRecording - Pausa la grabación. Puede ser reanudada al ejecutar el método record de nuevo.

stopRecording - Detiene la grabación, cierra el archivo y libera los recursos asociados. Tienes que llamar a este método al terminar de grabar; si no lo haces, el archivo con la grabación quedará inusable. Cmd-. al estar grabando tiene el mismo efecto.

recordNode - Devuelve al sintetizador que está llevando a cabo la grabación, para que pueda ser utilizado como objetivo. Esto sólo debe de ser necesario para los nodos que no estén creados en el grupo por defecto.



Los siguientes métodos "setter" tienen sus correspondientes "getters". Ve SoundFile para información acerca de los distintos tipos de formatos de muestra y de encabezado. No todos los formatos de muestra y de encabezado son compatibles.


recChannels_(anInteger) - Define el número de canales a grabar. El valor por defecto es dos. Tiene que ser llamado ANTES que prepareForRecord.


recHeaderFormat_(aString) - Define el formato de encabezado de la grabación. El valor por defecto es "aiff".  Tiene que ser llamado ANTES que prepareForRecord.

recSampleFormat_(aString) - Define el formato de muestra.  El valor por defecto es "float".  Tiene que ser llamado ANTES que prepareForRecord.


Toma en cuenta que la frecuencia de muestreo de la grabación será la misma que la de la aplicación servidor. Esto puede ser definido utilizando la instancia ServerOptions del servidor.



s.boot; // arranca el servidor


// algo que grabar

(

SynthDef("burbujas", {

var f, zout;

f = LFSaw.kr(0.4, 0, 24, LFSaw.kr([8,7.23], 0, 3, 80)).midicps; // función de glissando

zout = CombN.ar(SinOsc.ar(f, 0, 0.04), 0.2, 0.2, 4); // añadiendo ecos a una sinusoidal

Out.ar(0, zout);

}).send(s);

)


x = Synth.new("burbujas");


s.prepareForRecord; // tienes que llamar a este método primero


s.record;


s.pauseRecording; // se puede poner pausa


s.record // reanuda la grabación


s.stopRecording; // esto cierra el archivo y libera al nodo que llevó a cabo la grabación, etc.


x.free; // detén a los sintetizadores


// busca en tu carpeta de grabaciones y encontrarás un archivo que lleva por nombre la fecha y hora

// en que se llevó a cabo la grabación



Comandos Asíncronos


El servidor provee soporte para esperar la finalización de los comandos OSC asíncronos, como son el leer o escribir archivos de audio. N.B. Los siguientes métodos deben ser llamados desde una Routine que esté corriendo. Pasar una Condition explicitamente permite que múltiples elementos dependan de diferentes condiciones. Los ejemplos que se encuentran más abajo deben evidenciar la manera en que esto funciona.


bootSync(condition) - Arranca el servidor y espera hasta que haya acabado antes de reanudar las subrutina. El argumento condition es una instancia de Condition opcional utilizada para esta evaluación.


sendMsgSync(condition, args) - Envía el siguiente mensaje al servidor y espera hasta que haya acabado antes de reanudar la subrutina. El argumento condition es una instancia de Condition opcional utilizada para esta evaluación. El argumento arg debe de consistir de uno o más mensajes OSC válidos.


sync(condition, bundles, latency) - Envía un mensaje /sync al servidor, el cual contestará con el mensaje /synced cuando todos los comandos asíncronos pendientes hayan finalizado. El argumento condition es una instancia de Condition opcional utilizada para esta evaluación. En UDP dentro de una red de área amplia (WAN), este método puede ser un poco menos seguro que sendMsgSync, ya que los paquetes pueden llegar en desorden, pero en una red local debe de funcionar bien. En TCP, este método siempre debe de ser seguro. El argumento bundles debe de consistir de uno o más mensajes OSC que serán empaquetados antes del mensaje /sync (de esta forma se garantiza que llegarán antes que el mensaje /sync). El argumento latency permite que el mensaje sea evaluado en un punto específico en el futuro.


(

Routine.run {

var c;


// crea una instancia de la clase Condition para controlar la ejecución de la Routine

c = Condition.new;

s.bootSync(c);

\ARRANCADO.postln;

s.sendMsgSync(c, "/b_alloc", 0, 44100, 2);

s.sendMsgSync(c, "/b_alloc", 1, 44100, 2);

s.sendMsgSync(c, "/b_alloc", 2, 44100, 2);

\b_alloc_HECHO.postln;

};

)

c = Condition.new;

\ARRANCADO.postln;

s.sendMsg("/b_alloc", 1, 44100, 2);
