SynthDef representación de una definición de sintetizador del lado del cliente



superclass: Object

La aplicación del servidor utiliza las definiciones de sintetizadores como plantillas para crear nodos de Synth. (Los métodos como Function-play, etc. son simplemente métodos de conveniencia que automáticamente crean dichas definiciones.) La clase SynthDef encapsula la representación del lado del cliente de dicha definición, y ofrece métodos para crear nuevas definiciones, escribirlas en el disco, y transmitirlas a un servidor.

SynthDef es una de las clases más complicadas en SC y una explicación exhaustiva de ella está más allá de las miras de este documento. Como tal, los ejemplos más adelante y aquellos encontrados en los diversos tutoriales disponibles en Help podrán ser necesarios para aclarar algunos aspectos de su utilización.

Funciones de Graficación de UGen (UGen Graph Functions) y Formas Especiales de Argumentos

El núcleo de una definición es su función de graficación de generadores de unidades. Esta es una instancia de Function que detalla cómo los generadores de unidades de la definición están interconectados, sus entradas y salidas, y qué parámetros están disponibles para control externo. En un sintetizador basado en una definición, los argumentos de la función serán instancias de Control. Éstas podrán tener valores por defecto, o podrán ser establecidas en el momento de creación de un sintetizador. Después de la creación serán controlables a través de los métodos set y setn de Nodo (Node), o los mensajes OSC n_set y n_setn.

Hay cuatro tipos especiales de argumentos, que son tratados de maneras diferentes:

velocidad de audio (audio rate) - Los argumentos que empiecen con "a_" (p.ej. a_input), o que estén especificados como \ar en el argumento de velocidades de la definición (ver debajo), podrán leer de un bus de velocidad de audio al ser asignados a él mediante /n_mapa.

velocidad inicial (initial rate) - Los argumentos que empiecen con "i_" (p. ej. f_freq), o que estén especificados como \ir en el argumento de velocidades de la definición (ver debajo), serán estáticos y no modulables. No responderán a /n_set o /n_map. Esto es ligeramente más eficiente en términos de utilización del CPU que un argumento regular.

velocidad de trigger (disparo) (trigger rate) - Los argumentos que empiecen con "t_" (p. ej. t_trig), o que estén especificados como \tr en el argumento de velocidades de la definición (ver debajo), serán creados como un TrigControl. Asignar el argumento creará un impulso del valor elegido a velocidad de control. Esto es útil para los triggers.

arreglos literales (literal arrays) - Los argumentos que tengan arreglos literales como valores por defecto (ver Literals) resultan en controles multicanal, que pueden ser modificados en grupo con Node-setn o n_setn. Al editar dichos controles, no se verificará el límite del arreglo, así que el usuario tiene la responsabilidad de cerciorarse de que edite el número correcto de argumentos.

Ve los ejemplos debajo para más detalles sobre cómo funciona esto.

Ciertos nombres de argumentos (como 'out' para especificar un bus de salida) son tan comúnmente utilizados que adoptarlos puede ser considerado como un 'buen estilo'. Uno de éstos, 'gate', al ser usado para controlar la entrada de compuerta de un EnvGen, merece una mención especial, ya que permite al usuario utilizar el método release (liberar) de un Nodo. Ver Node para una explicación y más detalles.

Elementos estáticos versus dinámicos

Es importante entender que, a pesar de que una única definición puede proveer una gran flexibilidad a través de sus argumentos, etc., es, sin embargo, una unidad estática. La función de graficación de un UGen (y el código de SC detrás de ella) es evaluada únicamente cuando la definición es creada. De este modo, la utilización de enunciados 'if', etc., no tendrán ningún efecto en el momento que la definición es usada para crear un Synth, y es importante entender que la función de graficación de un UGen no deberá ser diseñada de la misma manera que las funciones en el lenguaje, donde múltiples evaluaciones pueden entregar resultados distintos. Será evaluada únicamente una vez.

Hay otras formas de obtener resultados similares, sin embargo, a menudo utilizando UGens como Rand. Por ejemplo, la siguiente definición tendrá una única frecuencia generada aleatoriamente, que será utilizada para cada Synth basado en ella:



(

SynthDef(\help_notRand, { Out.ar(0, SinOsc.ar(rrand(400, 800), 0, 0.2) 

* Line.kr(1, 0, 1, doneAction: 2)); }).send(s);

)

a = Synth(\help_notRand);

b = Synth(\help_notRand); // la misma frecuencia que a



Ésta, por otra parte, tendrá una frecuencia aleatoria diferente para cada Synth creado:

(

SynthDef(\help_isRand, { Out.ar(0, SinOsc.ar(Rand(400, 800), 0, 0.2) 

* Line.kr(1, 0, 1, doneAction: 2)); }).send(s);

)

a = Synth(\help_isRand);

b = Synth(\help_isRand); // una frecuencia elegida aleatoriamente

)



Métodos de la Clase

*new(name, ugenGraphFunc, rates, prependArgs, variants, metadata)

Crear una instancia de un SynthDef, evaluar la ugenGraphFunc y construir la ugenGraph.

name - Una cadena de caracteres (String) o símbolo (Symbol) (p. ej. "nombre" o \nombre). Este nombre será utilizado para referirse a la SynthDef al crear un Synth basado en ella y debe ser único.

ugenGraphFunc (función de graficación de los generadores de unidades) - Una instancia de Function especificando como los UGens de la definición están interconectados. Ver la discusión arriba para información de cómo los argumentos de la función son especificados.

rates - (velocidades) - Un arreglo opcional de especificaciones para los argumentos de la ugenGraphFunc. El orden corresponde al orden de los argumentos. Ver los ejemplos de bajo para observar cómo son usadas.

Una especificación puede ser:

nil/cero Un Control estándard a velocidad de control es creado.

\ar Un AudioControl a velocidad de audio es creado.

un número con punto decimal - el Control tendrá un retardo de la duración del tiempo especificado. Esto podrá ser usado para crear transiciones suaves entre diferentes valores. Los argumentos t_ y i_ no podrán ser retardos.

\ir El Control podrá ser asignado sólo al crear el Synth ('initial rate'). Ver la discusión arriba.

\tr El Control es usado como un trigger. Ver la discusión arriba.

prependArgs - Un arreglo opcional de objetos que serán pasados como los primeros argumentos a la ugenGraphFunc cuando sea evaluada. Los argumentos que reciban valores de este modo no serán convertidos a instancias de Control Ver el ejemplo de *wrap debajo para un ejemplo de cómo puede ser utilizado esto.

variants - Un Event opcional que contiene propiedades de argumentos por defecto. Podrá ignorar los valores por defecto especificados en la ugenGraphFunc. Al crear un Synth, una variante podrá ser solicitada añadiendo al argumento del nombre de la definición en la siguiente forma 'nombre.variante' o "nombre.variante". Ver el ejemplo debajo.

metadata - Un Event opcional conteniendo información adicional, definida por el usuario, que es relevante para el uso del SynthDef enel cliente. El SynthDef en sí es enviado al servidor para la generación del sonido; la metadata es estrictamente información descriptiva del lado del cliente. En este momento, el valor 'specs' en el evento está reservado para ControlSpecs a ser asociadas con los argumentos de una SynthDef (esto es útil para la construcción automáticamente de una GUI). La metadata puede ser almacenada en el disco y cargada automáticamente como parte de una SynthDesc. Ver el archivo de ayuda de SynthDesc para más detalles.

*writeOnce(name, ugenGraphFunc, rates, prependArgs, dir, metadata)

Crear una nueva SynthDef y escribirla al disco, pero sólo si el archivo de la definición no existe. Esto es útil en definiciones de clase para que la definición no sea escrita cada vez que la librería es compilada. Es importante notar que esto no revisará si existen diferencias, así que el usuario tendrá que eliminar el archivo de definición para que se vuelva a crear. Por defecto, el dir donde los archivos se almacenan está en la ruta especificada por SynthDef.synthDefDir.

*wrap(ugenGraphFunc, rates, prependArgs)

Envuelve una función dentro de una SynthDef contenedora. Los argumentos a ser contenidos son automáticamente convertidos a controles de SynthDef, utilizando las mismas reglas aplicadas a los argumentos de la función de UGen principal. Un ejemplo muy sencillo:



d = SynthDef(\demoWrapping, { |out|

Out.ar(out, SynthDef.wrap({ |freq| SinOsc.ar(freq) }))

});



d.allControlNames;

Nos muestra: [ ControlName  P 0 out control 0, ControlName  P 1 freq control 0 ]

La función externa declara el argumento 'out', y la función envuelta tiene 'freq' como su argumento. La SynthDef resultante tiene ambos argumentos y controles, exactamente como si la función externa incluyera ambos como sus argumentos.

El arreglo de velocidades (rates)se comporta como fue descrito anteriormente. PrependArgs permite que valores o generadores de unidades sean pasados a la función interior desde el contexto de SynthDef contenedor. Cualquier argumento interior que recibe un valor de prependArg (incluyendo nil) utilizará dicho valor, suprimiendo la creación de un control para dicho argumento. El ejemplo largo debajo ilustra esta técnica.

Esto es muy útil para producir SynthDefs en masa que tengan una "corteza" común definiendo características como los envolventes o desencadenando mecanismos que apliquen a distintas subgráficas de generadores de unidades. Las características comunes deberán ser escritas sólo una vez; los UGens que sean diferentes entre los SynthDefs son conectados en la estructura de soporte.

*synthDefDir

*synthDefDir_(dir)

Obtener o cambiar el directorio donde por defecto se escriben las definiciones.



Métodos de Instancia

writeDefFile(dir) 

Escribe la definición como un archivo llamado nombre.scsyndef de una forma legible para el servidor. Por defecto, dir es synthdefs/. Las definiciones almacenadas en dicho directorio serán cargadas automáticamente por los servidores interno y local al ser iniciados.

load(server, completionMessage, dir)

Escribir el archivo de definición y enviar un mensaje al servidor para cargar dicho archivo. Cuando este comando asincrónico es completado, el mensaje de completación (un mensaje OSC válido) es ejecutado inmediatamente por el servidor. El directorio por defecto para dir es synthdefs/.

send(server, completionMessage)

Compilar la definicón y enviarla al servidor sin escribir a disco (evitando de este modo la molesta acumulación de SynthDefs). Cuando este comando asincrónico es completado, el mensaje de completación (un mensaje OSC válido) es inmediatamente ejecutado por el servidor.

store(libname, dir, completionMessage, mdPlugin) 

Escribir el archivo de definición y guardarlo en la SynthDescLib especificada por libname, y enviar un mensaje al servidor de la libería para cargar dicho archivo. Cuando este comando asincrónico es completado, el mensaje de completación (un mensaje OSC válido) es inmediatamente ejecutado por el servidor. Por defecto, libname es \global, dir es synthdefs/. Esto se necesita para utilizar las definiciones con el sistema de transmisión de eventos. Ver Streams y Pattern.

mdPlugin (opcional) es la clase plug-in que será utilizada para almacenar la metadata. Si no es provista, el plug-in por defecto es utilizado. Ver el archivo de ayuda SynthDesc para más detalles.

memStore(libname, completionMsg)

Almacena la synthdef en la SynthDescLib especificada por libname, y la envía a los servidores de la librería. No se escribirá ningún archivo de definición; todas las operaciones ocurren en la memoria. Después de utilizar dicho método, la synthdef podrá ser usada con eventos de transmisión como en .store(), pero sin los efectos permanentes de un archivo en el disco.

Nota que los argumentos “dir” y “mdPlugin” no existen para este método. Debido a que no se escribe ningún archivo, no hay necesidad de especificar un directorio o un archivo de metadata.

play(target, args, addAction)

Un método de conveniencia que compila la definición y la envía al servidor destino (target). Cuando este comando asincrónico es completado, creará un sintetizador a partir de esta definicón, utilizando los valores de argumentos especificados en el arreglo args. Entrega un objeto Synth correspondiente. Para una lista de acciones de agregado ver Synth. Por defecto es \addToHead.

name

Entrega el nombre de esta definición.

variants

Entrega un Event conteniendo las variantes de esta definición.

Ejemplos

Básico

// Nota que construcciones como SynthDef(...) y Synth(...) son abreviaciones para SynthDef.new(...), etc.

// Con SynthDef es común conectar esto con llamadas a la instancia resultante, p. ej.SynthDef(...).send(s) or SynthDef(...).play

// crear una definición simple y enviarla al servidor



s.boot;

SynthDef(\SimpleSine, {|freq = 440| Out.ar(0, SinOsc.ar(freq, 0, 0.2)) }).send(s);



// lo anterior es esencialmente lo mismo que lo siguiente

d = SynthDef.new(\SimpleSine, {|freq = 440| Out.ar(0, SinOsc.ar(freq, 0, 0.2)) });

d.send(s);



// ahora crea un Synth a partir de la definición, utilizando los valores por defecto de freq, y luego otro con otro valor

x = Synth(\SimpleSine);

y = Synth(\SimpleSine, [\freq, 660]);



// ahroa cambia el valor de freq para x

x.set(\freq, 880);



x.free; y.free;



// utilizando el método de conveniencia 'play'

x = SynthDef(\SimpleSine, {|freq = 440| Out.ar(0, SinOsc.ar(freq, 0, 0.2)) }).play

x.free;





Argument Rates



// Las siguientes dos definiciones son equivalentes. La primera utiliza el un argumento 't_':

(

SynthDef(\trigTest, {|t_trig=0, freq=440| // t_trig create un TrigControl

Out.ar(0, SinOsc.ar(freq+[0,1], 0, Decay2.kr(t_trig, 0.005, 1.0)));

}, [0, 4] // retrasa la freq por 4 segundos (el segundo argumento), pero no t_trig (no funcionaría de todos modos)

);

)



// Esta segunda versión hace a trig un arg \tr al especificarlo en el arreglo de velocidades. Envía ésta.

(

SynthDef(\trigTest2, {|trig=0, freq=440| 

Out.ar(0, SinOsc.ar(freq+[0,1], 0, Decay2.kr(trig, 0.005, 1.0)));

}, [\tr, 4] // retrasa la freq (tiempo: 4s), \tr crea un TrigControl para trig

).send(s); 

)



// Utilizando la segunda versión para crear un synth

z = Synth.head(s, \trigTest2);



// ahora detona el envolvente de decaimienton

z.set(\trig, 1); // puedes hacer esto múltiples veces

z.set(\trig, 1, \freq, 220); // escucha cómo la frecuencia tiene un retraso

z.set(\trig, 1, \freq, 880);



z.free; //libera el sintetizador





Variantes

// crea una definición con algunas variantes

(

SynthDef(\vartest, {|out=0, freq=440, amp=0.2, a = 0.01, r = 1|

// el EnvGen con doneAction: 2 libera al sintetizador automáticamente al terminar

Out.ar(out, SinOsc.ar(freq, 0, EnvGen.kr(Env.perc(a, r, amp), doneAction: 2)));

}, variants: (alpha: [a: 0.5, r: 0.5], beta: [a: 3, r: 0.01], gamma: [a: 0.01, r: 4])

).send(s);

)



// ahora crea unos sintetizadores. Primero usando los argumentos por defecto.

Synth(\vartest);



// Ahora con las variantes:

Synth('vartest.alpha');

Synth('vartest.beta');

Synth('vartest.gamma');



// ignora una variante:

Synth('vartest.alpha', [\release, 3, \freq, 660]);





Arreglos Literales como Argumentos



// freqs tiene un arreglo literal de valores. Esto crea un control multicanal del mismo tamaño.

(

SynthDef(\arrayarg, { | amp = 0.1, freqs = #[300, 400, 500, 600], gate = 1 |

var env, sines;

env = Linen.kr(gate, 0.1, 1, 1, 2) * amp;

sines = SinOsc.ar(freqs +.t [0,0.5]).cubed.sum; // A mix of 4 oscillators

Out.ar(0, sines * env); 

}, [0, 0.1, 0]).send(s);

)



x = Synth(\arrayarg);

x.setn(\freqs, [440, 441, 442, 443]);



// No asignes demasiados valores, o podrás tener resultados inesperados

// El código siguiente inconscientemente cambia el argumento gate, y libera el sintetizador.

x.setn(\freqs, [300, 400, 500, 600, 0]);



// El ejemplo más complejo de Mr. McCartney

(

fork { 

z = Synth(\arrayarg);

2.wait;

10.do {

z.setn(\freqs, {exprand(200,800.0)} ! 4);

(2 ** (0..3).choose * 0.2).wait;

};



z.set(\amp, -40.dbamp);



10.do {

z.setn(\freqs, {exprand(200,800.0)} ! 4);

(2 ** (0..3).choose * 0.2).wait;

};

2.wait;

z.release;

};

)



Ejemplo de Wrapping: producción “en masa“ de definiciones de efectos

// La función makeEffect debajo envuelve una función más simple dentro de sí misma y provee

// una transición hacia el efecto (para que puedas añadirlo sin escuchar clicks), control de mezcla entre señales, etc.

// Dicha funcionalidad es útil para una variedad de efectos, y SynthDef.wrap te permite reutilizar código común

(

// el envolvente básico

~makeEffect = {| name, func, lags, numChannels = 2 |



SynthDef(name, {| i_bus = 0, gate = 1, wet = 1|

var in, out, env, lfo;

in = In.ar(i_bus, numChannels);

env = Linen.kr(gate, 2, 1, 2, 2); // fade in the effect

// llama al a función envuelta. Los argumentos in y env son pasados a la función como

// los primeros dos argumentos (prependArgs). 

// Cualquier otro argumento de la función envuelta será un Control.

out = SynthDef.wrap(func, lags, [in, env]);

XOut.ar(i_bus, wet * env, out);

}, [0, 0, 0.1] ).send(s);



};

)



// ahora crea un wah

(

~makeEffect.value(\wah, {|in, env, rate = 0.7, ffreq = 1200, depth = 0.8, rq = 0.1|

// in y env provienen del envolvente. El resto son controles

  var lfo;

lfo = LFNoise1.kr(rate, depth * ffreq, ffreq);

RLPF.ar(in, lfo, rq, 10).distort * 0.15; },

[0.1, 0.1, 0.1, 0.1],  // retrasos para la velocidad de ffreq, depth y rq

2 // numChannels

);

)



// ahora un reverb simple

(

~makeEffect.value(\reverb, {|in, env|

// in y env vienen del envolvente.

var input;

input = in;

16.do({ input = AllpassC.ar(input, 0.04, Rand(0.001,0.04), 3)});

input; },

nil,  // sin retraso

2 // numChannels

);

)



// algo para procesar

x = { {Decay2.ar(Dust2.ar(3), mul: PinkNoise.ar(0.2))} ! 2}.play;



y = Synth.tail(s, \wah);

z = Synth.tail(s, \reverb, [\wet, 0.5]);



// usamos un argumento llamado gate, para que Node-release pueda realizar las transiciones entre los efectos

y.release; 



// cambiar el valor de gate a 0 tiene el mismo resultado

z.set(\gate, 0);



x.free;



common argument names: out and gate



// los argumentos llamados 'out' y 'gate' son usados comúnmente para especificar un bus de salida y la compuerta de EnvGen respectivamente. Aunque no son requeridos, usarlos puede crear consistencia y intercambiabilidad 'gate' es particularmente útil, ya que permite el método .release de Node.

// método.

(

SynthDef(\synthDefTest, {|out, gate=1, freq=440|

// doneAction: 2 libera al Synth una vez que EnvGen termina

Out.ar(out, SinOsc.ar(freq) * EnvGen.kr(Env.asr(0.1, 0.3, 1.3), gate, doneAction:2));

}).store; // utilizaremos store para compatibilidad con el ejemplo de patrones debajo

)



x = Synth(\synthDefTest, [\out, 0]); // tocar el sintetizador a través del bus de salida 0 de hardware (ver Out.help)

x.release; // Libera y elimina el sintetizador (si doneAction es > 2; ver EnvGen)



//equivalente:



x = Synth(\synthDefTest); // el valor por defecto de out es 0, si no se le proporciona una argumento.

x.set(\gate, 0);



// si el valor es negativo, asigna el tiempo de liberacion, a gate-1

x = Synth(\synthDefTest);

x.set(\gate, -5); // liberación de 4 segundos



//equivalente:

x = Synth(\synthDefTest);

x.release(4);



// Si el argumento arg es usado de manera estándar, siempre puede ser cambiado sin conocer la synthdef

x = Synth(\synthDefTest, [\out, 0]);

x.set(\out, 1); //tocar a través del canal 1

x.release;
